## 1. JPA - `@EnableJpaAuditing` 누락 시 `@CreatedDate` 동작 안 함
`@CreatedDate`, `@LastModifiedDate`를 사용하려면 메인 애플리케이션 클래스에 `@EnableJpaAuditing`이 반드시 있어야 한다.
엔티티에 `@EntityListeners(AuditingEntityListener.class)`만 달아도 JPA Auditing을 활성화하는 설정이 없으면 동작하지 않는다.
```java
@EnableJpaAuditing
@SpringBootApplication
public class DakerApplication { ... }
```

## 2. JPA - `@UniqueConstraint`로 DB 레벨 중복 방지
서비스 계층에서 `existsByHackathonIdAndTeamId`로 중복을 검사하더라도, 동시 요청 등 예외 상황을 막으려면 DB 레벨에서도 제약을 걸어야 한다.
```java
@Table(
    uniqueConstraints = @UniqueConstraint(columnNames = {"hackathon_id", "team_id"})
)
```
서비스 레이어 검사 + DB 제약을 이중으로 적용하는 것이 안전하다.

## 3. JPQL - 엔티티 연관관계를 `.` 체이닝으로 탐색
`TeamMember → Team → Hackathon` 처럼 여러 단계 연관관계를 JPQL에서 JOIN 없이 경로 표현식(`.`)으로 탐색할 수 있다.
```java
@Query("SELECT tm FROM TeamMember tm WHERE tm.team.hackathon.id = :hackathonId AND tm.user.id = :userId")
Optional<TeamMember> findByHackathonIdAndUserId(...);
```

로컬 개발 환경에서 `Public Key Retrieval is not allowed` 오류가 발생하면 JDBC URL에 파라미터를 추가한다.

## 5. Spring 아키텍처 - 전체 흐름과 폴더 구조 파악 (AI 코드 생성 활용)
실제 코드는 Claude Code로 생성했기 때문에 직접 작성한 것은 아니지만, 생성된 코드를 읽으면서 Spring Boot 프로젝트가 어떤 구조로 돌아가는지 흐름을 파악했다.

**이해한 내용:**
- **Controller**: 클라이언트의 HTTP 요청을 받아서 Service에 넘기고, 결과를 응답으로 돌려주는 진입점.
- **Service**: 실제 비즈니스 로직이 들어가는 곳. 권한 검사, 상태 검사, 중복 방지 같은 판단을 여기서 한다.
- **Repository**: DB에 접근하는 계층. JPA가 인터페이스만 정의하면 구현체를 자동으로 만들어준다.
- **Domain(Entity)**: DB 테이블과 매핑되는 클래스. 단순 데이터 홀더가 아니라 `isRegistrationOpen()` 같은 도메인 판단 메서드도 가질 수 있다.
- **DTO**: 계층 간 데이터를 주고받을 때 쓰는 객체. 엔티티를 직접 외부에 노출하지 않기 위해 분리한다.

직접 구현하지는 않았지만 요청 → Controller → Service → Repository → DB 의 흐름과 각 계층이 왜 분리되어 있는지 감은 잡았다.

## 6. stats 도메인 - 별도 테이블 없이 Repository 조합으로 구현
집계성 API는 항상 새 테이블이 필요하지 않다. 기존 Repository를 조합하면 충분한 경우가 많다.
```java
// StatsService
long totalParticipants = userRepository.count();
long activeHackathons = hackathonRepository.countByStatusAndDeletedFalse(HackathonStatus.OPEN);
long totalPrizeKRW = prizeRepository.sumAllAmounts(); // COALESCE(SUM(amount), 0)
```
`PrizeRepository`의 `sumAllAmounts()`는 데이터가 없을 때 null 반환을 막기 위해 `COALESCE`를 사용한다.
```java
@Query("SELECT COALESCE(SUM(p.amount), 0) FROM Prize p")
long sumAllAmounts();
```

## 8. bash에서 `!` 문자 이스케이프 문제
bash에서 JSON 문자열에 `!`가 포함되면 히스토리 확장으로 오작동한다.
```bash
# 문제
curl -d '{"password":"Test123!"}'
# 해결 1: 단따옴표 내 !는 이스케이프 불가 → 비밀번호에서 ! 제거
# 해결 2: 쌍따옴표 + 이스케이프
curl -d "{\"password\":\"Test123\!\"}"
```

## 10. DB 스키마 설계 - 마스터 데이터와 인스턴스 데이터 분리
Java 콘솔 게임의 DB를 설계하면서, 게임 전체에서 공유되는 **마스터 데이터**와 세이브마다 달라지는 **인스턴스 데이터**를 분리하는 것이 중요하다는 걸 배웠다.

예를 들어 카드 시스템에서:
- `CARD` 테이블: 카드의 이름, 설명, 효과 등 **정의** (마스터 데이터)
- `SAVE_CARD` 테이블: 특정 세이브 슬롯이 어떤 카드를 보유하고 있는지 (인스턴스 데이터)

이렇게 분리하면 카드 정보를 한 곳에서 관리하면서도 세이브마다 독립적인 소지 상태를 유지할 수 있다. 컬렉션(도감, 업적, 엔딩)처럼 세이브와 무관하게 누적되어야 하는 데이터는 플레이어 단위로 연결해서, 세이브를 지워도 해금 기록이 유지되도록 설계했다.

## 11. DB 설계 - 다형성 FK와 실용적 비정규화
맵 시스템을 설계할 때, `stage_cell` 테이블에서 하나의 셀이 몬스터/NPC/아이템 중 하나를 참조해야 하는 상황이 생겼다. 이때 두 가지 선택지가 있었다:

1. **정규화 방식**: `stage_cell_monster`, `stage_cell_npc`, `stage_cell_item` 테이블을 각각 만든다.
2. **다형성 FK 방식**: `encounter_type`(MONSTER/NPC/ITEM)과 `encounter_ref_id` 컬럼 하나로 처리한다.

칸당 만남이 하나뿐인 이 구조에서는 테이블 3개로 쪼개는 것보다 다형성 FK가 더 실용적이었다. 정규화 원칙과 실용성 사이에서 상황에 맞게 판단하는 것도 설계의 일부라는 점을 배웠다.

## 12. MySQL - `CHECK` 제약과 `ON DELETE CASCADE` 활용
스키마 설계 과정에서 데이터 무결성을 지키기 위한 제약 조건들을 적용했다.

**CHECK 제약**: 열거형 값을 DB 레벨에서 제한할 수 있다.
```sql
game_status VARCHAR(20) DEFAULT 'PLAYING'
    CHECK (game_status IN ('PLAYING', 'CLEAR', 'GAMEOVER'))
```

**ON DELETE CASCADE**: 부모 행이 삭제되면 자식 행도 같이 삭제된다. 세이브 슬롯이 삭제될 때 해당 세이브의 소지 카드, 셀 상태 등이 함께 정리되어야 하므로 적용했다.
```sql
FOREIGN KEY (save_id) REFERENCES save_slot(save_id) ON DELETE CASCADE
```

**UNIQUE 복합 제약**: 같은 스테이지 같은 좌표에 셀이 중복 생성되는 것을 방지한다.
```sql
UNIQUE (stage_id, pos_x, pos_y)
```

## 13. 게임 서비스 플로우 설계 - 이중 루프 구조
게임 루프를 설계하면서, 전체 구조가 **이중 루프**로 이루어진다는 것을 이해했다.

- **외부 루프 (게임 루프)**: 맵 출력 → 입력 → 검증 → 이동 → 인카운터 처리를 반복한다.
- **내부 루프 (전투 루프)**: 몬스터를 만났을 때 턴제 전투가 별도 루프로 돈다.

게임오버 시에는 바로 이전 세이브를 로드할 수 있는 분기를 제공하고, 클리어 시에는 엔딩 조건(특정 카드 보유, 전 셀 방문 여부, NPC 대화 선택지 등)에 따라 노멀/히든/트루 엔딩을 판정하는 구조다. 이런 분기 설계를 미리 플로우로 정리해두면 실제 코딩할 때 훨씬 수월하겠다는 걸 느꼈다.

## 14. 클래스 다이어그램 - 레이어드 아키텍처 (게임 버전)
Spring 프로젝트와 유사하게, 콘솔 게임에서도 계층을 나눠 설계할 수 있다.

| 계층 | 역할 | 예시 |
|---|---|---|
| View | 콘솔 입출력 (Scanner, System.out) | MainMenuView, GameView |
| Service(Manager) | 게임 로직 조율 | GameManager, BattleManager, SaveManager |
| DAO | DB 접근 (JDBC) | PlayerDAO, SaveDAO, CardDAO |
| Model(Entity) | 데이터 + 도메인 로직 | Card.use(Monster), Monster.takeDamage() |
| Utility | 공통 인프라 | DBConnection |

GameManager가 다른 Manager들을 오케스트라처럼 조율하는 중심 역할을 하고, 각 Manager는 필요한 DAO를 호출하는 구조다. 이 설계 자체는 AI가 해준 것이지만, Spring의 Controller-Service-Repository 구조와 비교하면서 계층 분리가 왜 필요한지 이해하는 데 도움이 됐다.

## 15. DB 스키마 확장 - 기능 추가 시 기존 구조에 미치는 영향 파악
처음에 스테이지별 몬스터만 배치하는 단순한 구조(`stage_monster`)로 설계했다가, 3x3 그리드 맵 시스템을 추가하면서 스키마를 꽤 크게 손봤다.

**변경 사항:**
- `stage_monster` 삭제 → `stage_cell`로 대체 (몬스터 배치가 스테이지 단위 → 셀 단위)
- `stage`에 `map_width`, `map_height` 추가
- `save_slot`에 `current_x`, `current_y` 추가
- `npc`, `save_cell_state` 테이블 신설

하나의 기능(맵 이동)을 추가하면 관련 테이블이 연쇄적으로 영향을 받는다는 걸 체감했다. 처음부터 확장 가능성을 어느 정도 고려하되, 모든 걸 미리 설계하려고 하기보다는 필요할 때 구조를 발전시키는 게 현실적이라는 점도 느꼈다.
